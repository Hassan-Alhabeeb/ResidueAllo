RESEARCH PLAN — Updated after Phase 0 experiments
============================================================

ORIGINAL IDEA (TESTED & INVALIDATED):
Docking is slow because it simulates how a drug fits into a protein's
binding pocket by calculating interactions with EVERY amino acid.
The idea was to identify which residues matter, remove the rest,
and dock against a simplified pocket for speed.

PHASE 0 RESULT: Removing residues does NOT speed up Vina.
  - Full pocket:       4.99s, score -7.687 kcal/mol
  - Simplified pocket: 4.84s, score -7.737 kcal/mol
  - Speed difference:  3% (noise)
  - Vina's bottleneck is ligand conformational search, not receptor atoms
  - Removing residues HURT pose quality (less clustering)
  - Selective flexible docking pivot also rejected (field has moved past Vina)

--------------------------------------------------------------
PROJECT A: Membrane-Dependent Drug Access to COX-2
           (Higher impact, harder, 10-12 months)
--------------------------------------------------------------

TITLE: "Cholesterol-Dependent Modulation of NSAID Access to COX-2:
        A Molecular Dynamics Study at Physiological and Hyperthermic
        Temperatures"

WHY THIS WORKS:
  - COX-2 active site is accessed FROM INSIDE the membrane via a
    hydrophobic channel — membrane composition directly controls
    drug access
  - Cancer cells have higher cholesterol = thicker, more rigid
    membranes = harder for drugs to penetrate
  - Hyperthermia (42°C) disrupts membrane order
  - 2025 paper: cholesterol is a "heat shield" for cancer cells —
    deplete it and they become vulnerable
  - 2024 paper: ibuprofen gets captured by lipid rafts (cholesterol-
    rich domains where COX-2 sits)
  - NOBODY has put all pieces together computationally — genuine gap

THE GAP: No computational study integrating temperature + cholesterol
+ specific drug-target (COX-2) + cancer membrane context

STEP-BY-STEP PLAN:

  Month 1-2: Learning & Setup
    1. Install GROMACS (WSL2 on Windows works)
    2. Complete GROMACS membrane protein tutorial
    3. Learn CHARMM-GUI Membrane Builder (web-based)
    4. Build 4 membrane systems via CHARMM-GUI:
       - System A: POPC only (normal-like, low cholesterol)
       - System B: POPC + 30% cholesterol (normal cell)
       - System C: POPC + 40% cholesterol (cancer cell-like)
       - System D: POPC + sphingomyelin + 40% cholesterol (lipid raft)
    5. Insert COX-2 (4PH9) into each membrane using OPM orientation

  Month 3-4: Equilibration & Production MD
    6. Equilibrate all 4 systems at 310K (37°C)
    7. Duplicate all 4 and set thermostat to 315K (42°C)
    8. Run 500 ns production MD for each of the 8 systems
    9. Validate: area per lipid, membrane thickness, lipid order
       parameters — compare to published experimental values

  Month 5-7: Drug Permeation Analysis
    10. Umbrella sampling for ibuprofen permeation through each
        membrane (without COX-2 first — simpler system)
    11. Calculate PMFs (potential of mean force) for ibuprofen
        crossing each composition at both temperatures
    12. Result: 8 PMF profiles showing how energy barrier changes

  Month 7-9: COX-2 Active Site Access Analysis
    13. Analyze from 8 production trajectories:
        - Hydrophobic channel opening dynamics
        - Solvent accessibility of active site from membrane side
        - Membrane thickness/curvature near COX-2
    14. Dock ibuprofen into representative COX-2 conformations
        extracted from each trajectory (Vina skills transfer here)
    15. Compare binding poses/scores across membrane conditions

  Month 9-10: Integration & Writing
    16. Correlate: lower permeation barrier + more open channel
        = better drug access
    17. Identify which condition maximizes drug access
    18. Connect to cancer biology: does cancer-like membrane +
        hyperthermia specifically enhance NSAID access?
    19. Write and submit

TOOLS NEEDED:
  - GROMACS (free, GPU-accelerated)     — MUST LEARN
  - CHARMM-GUI (free, web-based)        — MUST LEARN
  - PLUMED (enhanced sampling)          — MUST LEARN
  - MDAnalysis (Python trajectory analysis) — MUST LEARN
  - PyMOL, Python, RDKit, Vina          — ALREADY HAVE

HARDWARE:
  - Need RTX 3080 or better (Linux/WSL2)
  - 4-8 months of continuous GPU time
  - Coarse-grained (Martini) approach could reduce to 2-4 weeks

RISKS:
  - 3-6 month learning curve for MD (biggest risk)
  - PMF convergence is notoriously difficult
  - Force field accuracy at 42°C less validated
  - Reviewers may demand experimental validation
  - Scope creep — must stay focused on 4 compositions × 2 temps

PUBLISHABILITY:
  - Target: J. Phys. Chem. B, JCTC, JCIM, or Membranes
  - Novelty: genuine gap (integration angle)
  - Clinical relevance: hyperthermia + cancer therapy framing
  - Low scooping risk (niche)

KEY REFERENCES:
  - Mizogami et al. (2025) — NSAID lipid raft interactivity + COX-2 selectivity
  - Golysheva et al. (2024) — Ibuprofen in raft-containing membranes
  - Cholesterol as cancer cell heat shield (2025, EurekAlert)
  - Boggara & Krishnamoorti (2010) — NSAID partitioning in bilayers via MD

PROJECT B: Allosteric/Cryptic Binding Site Detection
           (Safer, faster, 6-10 months)
--------------------------------------------------------------

TITLE: "ML-Based Prediction of Allosteric Binding Sites Using
        Structural and Evolutionary Per-Residue Features"

WHY THIS WORKS:
  - Allosteric drugs target "undruggable" proteins — major frontier
  - Same core skill as original plan: per-residue classification
    using conservation, B-factors, flexibility, contact density
  - Training data exists: AlloSteric Database (ASD)
  - Directly reuses existing Python/ML/structural biology skills
  - Not being made obsolete by new paradigms

THE GAP: Combining protein language model embeddings (ESM-2)
with structural nanoenvironment descriptors for allosteric site
prediction. Existing tools (PASSer, FPocket) use either sequence
OR structure, rarely both deeply integrated.

STEP-BY-STEP PLAN:

  Month 1-2: Data Collection & Feature Engineering
    1. Download AlloSteric Database (ASD) — curated allosteric
       sites with PDB structures
    2. Download negative examples from PDBBind (orthosteric-only
       proteins)
    3. Per-residue feature extraction:
       - B-factors (from PDB)
       - Conservation scores (ConSurf)
       - Solvent accessible surface area
       - Contact density (# neighbors at various cutoffs)
       - Secondary structure (DSSP)
       - Rotamer entropy
       - Distance to known orthosteric site
       - ESM-2 embeddings (per-residue, 1280-dim)

  Month 3-4: Model Development
    4. Binary classification: allosteric site residue (yes/no)
    5. Start simple: Random Forest / XGBoost on hand-crafted features
    6. Then: fine-tune ESM-2 or train a GNN on residue graphs
    7. Benchmark against existing tools:
       - PASSer2.0
       - FPocket
       - Allosite
       - SPACER
    8. Cross-validation across protein families (prevent data leakage)

  Month 5-7: Validation & Analysis
    9. Validate on held-out proteins with known allosteric sites
    10. Case studies on therapeutically relevant targets:
        - GPCRs with known allosteric modulators
        - Kinases with allosteric inhibitors (e.g., trametinib)
        - COX-2 (connects to our existing work)
    11. Feature importance analysis: which features matter most?
    12. Predict novel allosteric sites on proteins without known ones

  Month 7-9: Writing & Submission
    13. Write paper
    14. Create web tool or Python package for community use
    15. Submit to Bioinformatics, Briefings in Bioinformatics,
        or Communications Biology

TOOLS NEEDED:
  - Python, scikit-learn, PyTorch      — ALREADY HAVE / EASY
  - ESM-2 (protein language model)     — pip install
  - ConSurf (web server)               — free
  - DSSP (secondary structure)         — pip install
  - FPocket (pocket detection)         — free
  - PyMOL, RDKit                       — ALREADY HAVE

HARDWARE:
  - Any modern GPU for ESM-2 inference and GNN training
  - Much less compute than Project A
  - Could even run on Google Colab

RISKS:
  - Competitive field (PASSer, DeepAllo, STINGAllo exist)
  - Must show clear improvement over baselines
  - ASD dataset may have biases
  - Lower novelty than Project A

PUBLISHABILITY:
  - Target: Bioinformatics, Briefings in Bioinformatics, JCIM
  - Moderate novelty (method integration angle)
  - Higher scooping risk (active field, many groups)
  - Faster to publishable result

KEY REFERENCES:
  - PASSer2.0 (2022) — allosteric site prediction
  - STINGAllo (2024) — nanoenvironment descriptors
  - DeepAllo (2024) — PLM fine-tuning for allosteric residues
  - AlloSteric Database (ASD, 2024 update)
